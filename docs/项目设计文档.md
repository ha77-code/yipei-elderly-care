# “医陪”——银发人群陪诊与生活陪伴服务平台

**英文名称：** YiPei — Elderly Medical Escort and Daily Companionship Service Platform

## 1. 项目概述

### 1.1 项目定位

“医陪”是一个面向老年人及其子女的陪诊与生活陪伴服务平台，帮助子女远程为父母预约陪诊服务，也帮助老人获得安全、体面、易理解的就医和日常生活支持。

平台连接三类角色：

- 需要服务的老人及其家属；
- 提供陪诊、陪伴和代办服务的陪诊师；
- 负责审核、订单管理和服务监督的平台管理员。

### 1.2 项目类型

Web 应用类（全栈开发）+ 生活服务平台类 + AI 辅助工具类。

### 1.3 一句话定位

> 帮子女远程照料父母，帮老人安全、体面地就医和生活。

### 1.4 项目边界

本项目是陪诊服务信息发布、人员匹配和订单管理平台，不是医疗诊断平台。陪诊师只提供陪同、引导、手续协助、信息记录和生活陪伴，不进行诊断、治疗、用药决策或医疗操作。

## 2. 背景与用户痛点

随着老年人口增加，独居、空巢和子女异地工作的情况越来越普遍。老人前往医院时可能遇到以下困难：

- 不熟悉线上挂号、自助机和缴费流程；
- 医院空间复杂，难以快速找到科室和检查地点；
- 排队、取号、取药等流程耗时较长；
- 听不清或记不住医生的注意事项；
- 行动不便，独自就医存在安全风险；
- 子女因工作、距离或时间安排无法陪同；
- 老人日常生活中也可能需要聊天、散步、手机教学和代办支持。

传统解决方式主要依靠亲属、熟人介绍或临时寻找个人陪同人员，存在服务信息不透明、人员可信度不足、订单过程难以追踪和售后责任不清晰等问题。

## 3. 项目目标

### 3.1 平台目标

建立一个透明、可查询、可评价的陪诊服务信息平台，完成以下闭环：

```text
家属/老人发布需求
    ↓
平台展示陪诊师
    ↓
用户选择或申请陪诊师
    ↓
创建服务订单
    ↓
陪诊师接单并提供服务
    ↓
服务完成确认
    ↓
双方评价与平台记录
```

### 3.2 系统目标

- 为老人及家属提供清晰的服务需求发布入口；
- 为陪诊师提供资料展示和订单管理能力；
- 为平台管理员提供陪诊师审核、订单监督和投诉处理能力；
- 通过评分、服务次数和标签帮助用户选择陪诊师；
- 通过 AI 辅助整理服务需求，降低信息填写难度；
- 通过订单状态和服务记录提高服务过程的可追踪性。

### 3.3 不在项目范围内的功能

为了控制实训项目的复杂度，平台暂不实现：

- 真实微信或支付宝支付；
- 真实资金托管和自动提现；
- 医疗诊断、处方建议和用药决策；
- 陪诊师进行医疗操作；
- 实时 GPS 轨迹和电子围栏；
- 实时视频和 WebSocket 聊天；
- 医疗保险理赔；
- 短信验证码服务；
- 医院内部系统对接；
- 复杂的智能派单算法。

## 4. 用户角色与权限

### 4.1 老人/家属用户

- 注册和登录；
- 完善老人或家属资料；
- 发布陪诊或生活陪伴需求；
- 浏览陪诊师资料；
- 查看服务价格和可服务时间；
- 创建服务订单；
- 查看订单状态；
- 取消未开始的订单；
- 确认服务完成；
- 对陪诊师进行评价；
- 查看历史订单和服务记录。

家属可以代替老人发布需求，但需要在需求中填写被服务老人的基本情况和特殊注意事项。

### 4.2 陪诊师/陪伴师

- 提交入驻申请；
- 填写个人资料、服务区域和服务类型；
- 上传头像或资质说明材料；
- 查看可接订单；
- 接受或拒绝订单；
- 更新服务状态；
- 填写服务记录；
- 查看收入模拟数据；
- 查看用户评价。

陪诊师资料必须经过管理员审核后才能对外展示和接单。

### 4.3 平台管理员

- 审核陪诊师入驻申请；
- 管理用户和陪诊师资料；
- 管理服务类型、医院和科室基础数据；
- 查看和干预订单状态；
- 处理投诉和举报；
- 管理评价内容；
- 查看订单量、完成率和平台抽成模拟数据。

## 5. 服务类型设计

### 5.1 陪诊服务

- 诊前沟通；
- 医院路线和科室引导；
- 协助使用挂号机和缴费机；
- 协助排队、取号和取药；
- 记录医生说明和检查事项；
- 陪同老人安全离开医院；
- 将服务情况反馈给家属。

### 5.2 生活陪伴服务

- 陪聊天；
- 陪散步；
- 教授基础手机操作；
- 陪同参加社区活动；
- 代购日常用品；
- 协助办理非医疗类生活事务。

涉及代购或代办时，系统只记录服务需求和订单信息，实训版本不处理真实资金支付。

## 6. 核心业务流程

### 6.1 陪诊需求发布

用户填写：

- 服务类型；
- 服务日期和时间；
- 医院名称；
- 科室；
- 就诊事项；
- 老人行动情况；
- 是否需要轮椅或特殊协助；
- 联系人姓名和电话；
- 其他注意事项；
- 预算或服务价格档位。

系统可以根据文字描述调用 AI 生成需求摘要和服务标签，用户确认后保存。

### 6.2 陪诊师选择

系统展示陪诊师：

- 姓名或昵称；
- 服务类型；
- 服务区域；
- 擅长场景；
- 累计完成服务次数；
- 用户评分；
- 服务价格；
- 认证状态；
- 可预约时间。

首版采用简单规则排序：

```text
认证状态优先
    ↓
服务类型匹配
    ↓
服务区域匹配
    ↓
评分和完成次数排序
```

不实现实时 GPS 距离计算，区域使用省、市、区或手动填写的服务区域。

### 6.3 订单状态流转

订单状态统一定义为：

```text
待确认
    ↓
待接单
    ↓
已接单
    ↓
服务中
    ↓
待确认完成
    ↓
已完成
```

异常状态：

```text
已取消
已拒绝
已投诉
```

状态规则：

- 用户发布需求后为“待接单”；
- 陪诊师接受后为“已接单”；
- 服务开始时改为“服务中”；
- 陪诊师提交服务记录后为“待确认完成”；
- 用户确认后为“已完成”；
- 未开始服务的订单可以取消；
- 已完成订单不能被普通用户直接修改。

### 6.4 服务完成与评价

服务完成后，用户可以评价：

- 服务态度；
- 准时程度；
- 沟通情况；
- 服务是否完成；
- 文字评价。

陪诊师也可以对用户进行简单评价，例如：

- 需求描述是否准确；
- 是否按约定时间出现；
- 沟通是否顺畅。

评价只用于平台信用展示，不作为医疗判断依据。

## 7. AI 辅助功能

### 7.1 需求摘要生成

用户输入：

```text
下周三上午陪我母亲去人民医院心内科复诊，她行动比较慢，需要有人帮忙取号、带路和记录医生交代的事情。
```

AI 可以生成：

```text
服务类型：陪诊
时间：下周三上午
地点：人民医院
科室：心内科
特殊需求：行动较慢、需要路线引导、需要记录医嘱
```

### 7.2 陪诊师标签生成

根据陪诊师填写的资料生成：

- 医院陪诊；
- 老年人陪同；
- 行动不便陪同；
- 手续办理；
- 医嘱记录；
- 手机操作指导。

### 7.3 AI 使用边界

AI 只负责：

- 整理文字；
- 提取服务标签；
- 生成需求摘要；
- 辅助推荐服务类型。

AI 不负责：

- 医疗诊断；
- 判断病情；
- 推荐药物；
- 解释检查结果；
- 判断老人是否适合独自出行；
- 自动确认陪诊师资质；
- 自动完成订单审核。

AI 服务异常时，用户仍可以手动填写信息，不影响订单流程。

## 8. 商业价值

### 8.1 目标用户

- 独居老人和空巢老人；
- 无法长期陪同父母的异地子女；
- 行动不便但需要非医疗陪同的老人；
- 兼职陪诊师和生活陪伴师；
- 社区养老服务机构；
- 养老院和社区服务中心。

### 8.2 用户价值

对于老人：

- 获得更安全、更清晰的就医陪同；
- 减少独自操作医院设备的困难；
- 有人协助记录和转达服务过程；
- 获得日常生活陪伴。

对于子女：

- 可以远程发布和管理父母的服务需求；
- 可以查看订单状态和服务记录；
- 可以选择有审核记录和评价的陪诊师；
- 减少因工作和距离无法陪诊产生的焦虑。

对于陪诊师：

- 获得服务展示和订单来源；
- 建立个人服务记录和评价；
- 管理可服务时间和订单状态；
- 逐步形成个人服务信用。

对于机构：

- 统一管理陪诊和陪伴服务；
- 记录人员、订单和评价；
- 提高服务流程的透明度；
- 通过统计数据优化人员安排。

### 8.3 商业模式

平台可以采用“服务撮合 + 机构服务”的方式盈利：

1. **服务撮合抽成**：每笔服务订单按比例收取平台服务费；
2. **家属会员服务**：提供优先匹配、历史服务管理和专属客服等权益；
3. **陪诊师入驻服务**：为陪诊师或小型陪诊机构提供认证展示和订单管理；
4. **机构版 SaaS**：为养老机构、社区和服务中心提供人员、订单和评价管理后台；
5. **定制部署服务**：为机构提供独立部署、数据报表和品牌化页面。

实训系统中只记录模拟订单金额、平台抽成和陪诊师收入，不接入真实支付。

### 8.4 成本控制

- 订单和用户数据以文字、数字为主；
- 不依赖图片识别和地图实时定位；
- 不接入真实支付和短信服务；
- AI 只处理短文本，调用次数可控；
- 图片仅用于头像或资料说明，首版使用本地存储；
- 本地开发和演示不需要购买独立服务器；
- 后期使用单台普通云服务器即可部署原型。

## 9. 技术架构

### 9.1 技术栈

- **开发语言**：Java；
- **JDK**：JDK 21；
- **后端框架**：Spring Boot 3.x；
- **持久层**：MyBatis；
- **数据库**：MySQL 8；
- **缓存**：Redis；
- **前端**：Vue2 + ElementUI；
- **请求工具**：Axios；
- **AI 接口**：文本大模型 API，用于需求摘要和服务标签；

- **认证权限**：普通用户、陪诊师、管理员角色权限；
- **部署**：本地运行或单台普通云服务器。

### 9.2 系统架构

```text
Vue2 + ElementUI
        │ Axios
        ▼
Spring Boot Backend
        │
 ┌──────┼──────────────┐
 ▼      ▼              ▼
MySQL  Redis          AI API
        │
   Local File Storage
```

## 13. 安全与合规边界

- 平台明确声明不提供医疗诊断和治疗服务；
- 陪诊师不能修改处方、代替医生做医疗决定或进行医疗操作；
- 陪诊师入驻资料需要管理员审核；
- 用户联系方式只用于订单服务，不应公开展示；
- 订单中涉及的老人健康信息应尽量减少收集；
- 重要服务备注只能由订单相关人员查看；
- 所有订单状态变更记录操作人和时间；
- 投诉、取消和异常服务需要保留处理记录；
- 实训数据使用虚拟用户和虚拟医院，不使用真实个人健康信息；
- 真实支付、保险和医疗机构合作必须在具备合规条件后单独设计。

## 15. 项目验收标准

- 用户可以注册、登录并按照角色进入不同页面；
- 陪诊师可以提交资料，管理员可以审核；
- 用户可以发布陪诊需求；
- 用户可以浏览和筛选陪诊师；
- 用户可以创建服务订单；
- 陪诊师可以接单并更新服务状态；
- 用户可以确认服务完成；
- 双方可以创建评价；
- 管理员可以查看订单、审核资料和处理投诉；
- AI 可以根据文字生成需求摘要或服务标签，失败时支持手动填写；
- Redis 可以完成登录状态、缓存或防重复提交功能；
- 项目使用虚拟数据完成完整演示，不依赖真实支付和医院系统。

## 16. 项目总结

“医陪”以老年人就医和生活陪伴为应用场景，连接家属、老人、陪诊师和服务机构，提供需求发布、人员展示、订单管理、服务记录和评价监督等功能。

项目具有明显的社会价值和商业想象空间，同时通过模拟支付、手动区域、虚拟医院和文本 AI 的方式控制开发与部署成本，适合使用 Spring Boot、MyBatis、MySQL、Redis、Vue2 和 ElementUI 完成企业级实训开发。
